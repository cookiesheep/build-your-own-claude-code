# WORK_LOG.md

> 这是项目的工作留痕入口文档。
> 真正的权威工作日志在：`internal/TEAM_PROGRESS.md`

---

## 为什么需要这份文件

这个项目会频繁切换：

- 不同 AI agent
- 不同 thread / 会话
- 不同开发方向（前端 / 后端 / Lab / 文档 / 基础设施）

为了避免上下文断裂、重复调研、进度丢失，项目要求每次完成实质性工作后，都要留下清晰记录。

---

## 唯一权威日志

统一在以下文件记录工作：

- `internal/TEAM_PROGRESS.md`

不要再额外维护第二份独立的“正式工作日志”，否则很容易分叉。

---

## 每次工作后必须记录什么

每次工作结束后，必须在 `internal/TEAM_PROGRESS.md` 中追加一条带日期的记录，至少包含：

1. 完成项
2. 进行中
3. 阻塞项
4. 下一步建议
5. 关键验证方式

推荐格式：

```md
### YYYY-MM-DD（会话 N）

**完成项**：
- ✅ ...

**进行中**：
- 🔄 ...

**阻塞项**：
- ⚠️ ...

**下一步**：
- ...
```

更推荐使用下面这份标准模板，尽量不要自由发挥字段名：

```md
### YYYY-MM-DD（会话 N / 方向名）

**完成项**：
- ✅ 做了什么
- ✅ 改了哪些关键文件 / 模块

**进行中**：
- 🔄 还在推进什么

**阻塞项**：
- ⚠️ 被什么卡住
- ⚠️ 缺少什么前置条件 / 外部依赖

**验证**：
- `命令或验证方式`
- `命令或验证方式`

**下一步**：
- 继续做什么
- 建议下一个 agent 优先处理什么
```

---

## 什么时候还要更新别的文档

如果这次工作影响了项目整体认知，还要同步更新：

- `internal/PROJECT_BRIEFING.md`
  - 当项目阶段、优先级、平台方案、Lab 结构、默认阅读顺序发生明显变化时

- `SESSION_STARTER.md`
  - 当你希望未来新会话改变启动阅读顺序或工作方式时

- `HANDOFF.md`
  - 当发生重大架构决策、关键 PoC、重要结论变更时

---

## 新会话接手顺序

任何新 agent / 新会话开始工作前，优先阅读：

1. `WORK_LOG.md`
2. `internal/TEAM_PROGRESS.md`
3. `internal/PROJECT_BRIEFING.md`
4. 该方向对应的 `internal/work-*/AGENT_CONTEXT.md`

---

## 当前执行原则

- `internal/TEAM_PROGRESS.md` 是唯一权威工作日志
- `WORK_LOG.md` 是根目录入口与规则说明
- 所有 agent 完成实质性工作后都必须更新日志
- 不允许“做完事但不留痕”
- 留痕时优先使用本文件中的标准模板
