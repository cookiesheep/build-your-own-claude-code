# 后端个人笔记

**后端到底是在做什么**

在这个项目里，后端不是“再做一个网页”。后端是平台的“控制层”和“执行层”。

前端负责让学习者看到页面、写代码、点按钮、看终端。
后端负责把这些操作变成真实的系统行为：

- 给这个学习者分配一个独立容器
- 把他写的代码放进容器
- 在容器里触发构建
- 把构建结果返回给前端
- 把浏览器终端连接到容器里的真实 shell
- 记录这个学习者当前做到哪个 Lab

所以你可以把后端理解成一句话：

**前端负责交互，后端负责让“真实 Claude Code 运行起来”。**

------

**这个项目里后端的最终目标**

我们最终要做成的，不是几个 API 文档，而是一个完整闭环：

1. 学习者打开 Lab 页面
2. 平台自动给他准备一个运行环境
3. 学习者在 Monaco 里补代码
4. 点“提交代码”
5. 后端把代码注入到 Docker 容器
6. 后端在容器里执行 node build.mjs --lab
7. 前端拿到构建日志并展示
8. 学习者在浏览器终端里运行 node cli.js
9. 看到真实 Claude Code TUI 被自己的代码驱动

这就是后端真正要完成的事。

------

**为什么这个项目一定需要后端**

如果只是普通教程，其实前端 + 测试就够了。
但你们这个项目的核心卖点不是“测试变绿”，而是：

**学习者能在浏览器里看到真实 Claude Code TUI。**

这件事决定了必须有后端，因为：

- 浏览器不能安全地直接运行完整 Node.js + Docker + ttyd 链路
- 浏览器不能自己管理容器
- 浏览器不能直接注入代码到服务器环境
- 浏览器也不适合直接承担真实终端代理

所以后端不是附属品，而是这个项目价值主张的一部分。

------

**你以后在后端开发里，实际在做哪些模块**

我用“平台运营后台”的视角给你拆开。

**1. Session / 会话管理**
这是“谁在用平台”的最小身份层。

它要解决的问题是：

- 当前这个页面属于哪个学习者会话
- 这个会话有没有已经存在的容器
- 如果没有，要不要新建一个
- 如果有，这个容器现在还活着吗

在代码里，这一层主要对应：
[session.ts](app://-/index.html?hostId=local)
[database.ts](app://-/index.html?hostId=local)

你可以把它理解为：
**“给每个学习者发一个临时实验桌。”**

------

**2. 容器管理**
这是后端最核心的部分。

平台不能让所有学习者共用同一个运行目录，否则文件会互相污染。
所以每个会话需要一个自己的容器。

容器管理要做的事：

- 创建容器
- 启动容器
- 查容器状态
- 删除容器
- 重建容器
- 找到容器里 ttyd 对应的宿主机端口

对应文件是：
[container-manager.ts](app://-/index.html?hostId=local)

你可以把它理解为：
**“后端在后台替每个学习者开一台一次性小电脑。”**

------

**3. 代码提交与注入**
学习者在前端编辑器里写的是字符串。
后端要把这个字符串变成容器里的真实文件。

这一步会做：

- 接收 sessionId
- 接收 code
- 接收 labNumber
- 把代码写入容器里的目标文件
- 保证特殊字符不会把 shell 命令搞坏

对应文件是：
[submit.ts](app://-/index.html?hostId=local)

你可以把它理解为：
**“把学生在网页里写的代码，复制到那台实验电脑里。”**

------

**4. 构建触发**
代码写进去还不够，还要编译和替换进真实运行底座。

这一步会做：

- 在容器里执行 node build.mjs --lab
- 捕获 stdout / stderr
- 判断 exit code
- 告诉前端这次构建成功还是失败
- 把失败日志原样返回给学习者

这也是 [submit.ts](app://-/index.html?hostId=local) 和 [container-manager.ts](app://-/index.html?hostId=local) 的核心职责。

你可以把它理解为：
**“按下实验电脑里的编译按钮，并把编译器输出抄回来。”**

------

**5. 终端代理**
这是最容易让人迷糊，但其实概念很简单。

学习者在浏览器里看到的是 xterm.js。
xterm.js 不是 shell，它只是一个“终端外观组件”。

真实 shell 在容器里。
容器里用 ttyd 把 shell 暴露成 WebSocket 服务。
后端再把浏览器来的 WebSocket 连接转发到 ttyd。

链路是：

浏览器 xterm.js -> 后端 WebSocket 代理 -> 容器 ttyd -> bash/node cli.js

对应文件是：
[ws-proxy.ts](app://-/index.html?hostId=local)
[Terminal.tsx](app://-/index.html?hostId=local)

你可以把它理解为：
**“前端看到的是一个窗口，真正的终端在容器里，后端负责把这根线接上。”**

------

**6. 重置机制**
学习者可能把环境搞乱了，或者想重新开始。
后端要提供一个“回到初始状态”的能力。

最稳的 MVP 做法不是在容器里清理文件，而是：

- 删除旧容器
- 重新创建新容器

对应文件：
[reset.ts](app://-/index.html?hostId=local)

你可以把它理解为：
**“把实验桌整张换掉，而不是在旧桌子上慢慢擦。”**

------

**7. 进度存储**
平台还要知道这个学习者做到了哪一步。

这一步通常记录：

- sessionId
- labNumber
- 是否完成
- 完成时间

对应文件：
[database.ts](app://-/index.html?hostId=local)
[progress.ts](app://-/index.html?hostId=local)

你可以把它理解为：
**“老师的点名册和成绩表。”**

------

**后端最后到底会交付什么**

从工程产物看，后端最终不是一个抽象概念，而是这些具体东西：

- 一个 Node/Express 服务
- 一组 API 路由
- 一个 Docker 管理层
- 一个 WebSocket 代理层
- 一个 SQLite 数据库
- 一套容器镜像和运行约定

当前骨架就在这里：

- [index.ts](app://-/index.html?hostId=local)
- [database.ts](app://-/index.html?hostId=local)
- [container-manager.ts](app://-/index.html?hostId=local)
- [ws-proxy.ts](app://-/index.html?hostId=local)
- [session.ts](app://-/index.html?hostId=local)
- [submit.ts](app://-/index.html?hostId=local)
- [progress.ts](app://-/index.html?hostId=local)
- [reset.ts](app://-/index.html?hostId=local)

------

**你们真正要先做好的，不是什么**

现在最容易走偏的地方，是一上来把后端想成“很大的系统设计题”。

其实不是。

MVP 阶段最重要的不是：

- 完整用户系统
- 复杂权限系统
- 容器调度平台
- 高并发架构
- 很高级的数据库设计

最重要的是：

**打通最小闭环。**

也就是：

- 能创建会话
- 能创建容器
- 能提交代码
- 能触发构建
- 能看到构建日志
- 能把浏览器终端接到容器里

只要这 6 件事打通，平台就开始“活了”。

------

**如果你完全没有后端经验，你现在最应该形成的心智模型**

不要把后端想成“很神秘”。

你可以把整个后端理解成一个餐厅后厨：

- 前端是点餐窗口
- API 是服务员接单
- Session 是桌号
- Docker 容器是每桌独立的小厨房
- 代码注入是把菜单交给厨师
- 构建是开始做菜
- buildLog 是厨房回传的状态
- ttyd + WebSocket 是把厨房直播接到顾客面前
- SQLite 是账本

这个类比虽然粗，但非常有助于你不被术语吓住。

------

**你个人后面开发时，最像是在做什么**

你后面真正会做的，不是“研究 AI 原理”，而是做这些典型后端工作：

- 接收请求
- 校验参数
- 调用 Docker
- 执行命令
- 收集日志
- 返回 JSON
- 管理状态
- 处理失败情况
- 把系统不同部分连接起来

这就是标准的软件后端工程，只不过你们服务的对象不是普通 CRUD 页面，而是一个“代码教学 + 容器运行平台”。

------

**我建议你现在先记住的三句话**

- 后端的职责是让“真实 Claude Code 运行起来”，不是单纯返回几个接口。
- 当前阶段最重要的是平台最小闭环，不是大而全架构。
- 你们现在要学的不是“所有后端知识”，而是这个项目真正需要的那一小块后端能力。
